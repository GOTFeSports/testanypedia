# /admin-data — служебные данные Telegram-бота Anypedia

Эта директория не подключается фронтом сайта (`index.html`, `tournament.html` и т.д. её не знают). Здесь хранятся только данные, нужные Telegram-боту для модерации турниров, прав доступа и уведомлений. Источник истины для публичных данных сайта остаётся `data.js`, `teams.js`, `organizers.js` в корне репозитория.

## Файлы

### `drafts.json`
Черновики турниров, отправленные организаторами через бота, до прохождения модерации.

- `status`: `"pending"` | `"approved"` | `"rejected"` | `"needs_changes"`
- `payload` — объект с полями турнира в том же формате, что и элемент массива `tournaments[]` в `data.js`, но **без** `id` (присваивается ботом при approve, генерируется слагом из `title`) и без `bracket`/`teamsList` (заполняются отдельными шагами после публикации).
- При `approve` бот переносит `payload` (дополненный сгенерированным `id` и дефолтными пустыми `casters: []`, `links: {}`, `teamsList: []`, если организатор их не указал) в конец массива `tournaments[]` в `data.js`, и меняет статус черновика на `"approved"`. Запись в `drafts.json` не удаляется — остаётся как история.
- При `reject` статус меняется на `"rejected"`, `data.js` не трогается.

### `permissions.json`
Whitelist ролей.

- `role`: `"admin"` | `"organizer"` | `"captain"`
- Для `organizer` обязателен `organizerId`, соответствующий `id` записи в `organizers.js`.
- Для `captain` обязателен `teamId`, соответствующий `id` записи в `teams.js`.
- Любое действие бота, требующее прав (создание черновика, approve, обновление счёта матча), проверяется по этому файлу до выполнения.

### `subscriptions.json`
Подписки пользователей на уведомления о командах.

**Важно**: ключ подписки — `teamId` (стабильный `id` из `teams.js`), а не название команды. Названия и алиасы команд могут меняться, `id` нет — поэтому привязка идёт строго по нему, чтобы подписка не "терялась" при переименовании команды.

### `bot-state.json`
Текущие настройки и dedup-данные:

- `announceChatId` — куда бот постит анонсы новых опубликованных турниров.
- `featureFlags` — переключатели функциональности, чтобы включать этапы roadmap независимо без редеплоя кода (например, `liveMatchesEnabled: false` пока этап live-матчей не готов, даже если код уже задеплоен).
- `lastNotifiedHashes` — защита от дублирующих уведомлений при повторных коммитах/ретраях.

### `activity-log.json`
Append-only аудит-лог всех действий организаторов и администраторов через бота. Записи никогда не редактируются и не удаляются — только добавляются в конец `entries[]`. Используется для:

- истории, кто и когда создавал/одобрял/отклонял турниры
- истории изменений счёта матчей (кто и когда вносил)
- разбора спорных ситуаций ("организатор говорит, что не трогал счёт — лог покажет, было ли действие")

`action` — строка вида `"draft.created"`, `"draft.approved"`, `"draft.rejected"`, `"match.score_updated"`, `"bracket.created"`, `"roster.updated"` и т.д. — единый формат `сущность.событие`.

## Принцип работы бота с этими файлами

1. Бот читает нужный JSON через GitHub Contents API, парсит как обычный `JSON.parse` (это чистый JSON, в отличие от `data.js`, который JS-файл с `const tournaments = [...]`).
2. Вносит изменение в распарсенный объект.
3. Сериализует обратно (`JSON.stringify(obj, null, 2)`) и коммитит через тот же API.
4. Каждое действие, меняющее `drafts.json`, `permissions.json`, `subscriptions.json` или `data.js`, дополнительно дописывает запись в `activity-log.json` тем же коммитом или отдельным сразу после — для полной прослеживаемости.

Коммиты в `data.js` остаются текстовой вставкой в массив `tournaments[]` (см. комментарии в самом файле) — формат файла как JS, а не JSON, сохраняется без изменений, чтобы не ломать совместимость с тем, как сейчас вручную правится репозиторий.
